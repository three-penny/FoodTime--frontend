作者：XXXXX

适用项目：`FoodTime-frontend`

## 1. 功能范围

本次前端新增管理员审核台，用于审核用户侧提交的两类内容：

+ 菜品投稿：用户上传平台暂无的菜品，管理员通过后进入菜品展示候选池，驳回后用户可看到处理意见。
+ 吐槽墙内容：用户提交吐槽后先进入待审核状态，通过后展示到吐槽墙，驳回后不公开展示。

当前前端仍使用 Pinia + mock 数据模拟审核流，后端接入时建议在 `src/api/` 下封装请求，再由 store 调用。

## 2. 前端页面与路由

```text
/admin/audit    管理员审核台
```

路由名称：`adminAudit`

权限规则：

+ 必须登录：`meta.requiresAuth = true`
+ 必须为管理员：`meta.requiresAdmin = true`
+ 前端当前通过 `useAuthStore.currentRole === 'admin'` 判断管理员身份

## 3. 状态定义

菜品投稿和吐槽墙内容统一使用以下审核状态：

```text
pending   待审核
approved  已通过
rejected  已驳回
```

建议后端所有审核类接口统一返回 `status` 和 `reason` 字段。`reason` 用于展示管理员处理意见，通过时可为空或返回通过说明。

## 4. 管理员登录建议

前端当前管理员登录仅做本地模拟。后端接入时建议提供：

### POST /api/auth/login

请求：

```json
{
  "account": "admin001",
  "password": "******",
  "role": "admin"
}
```

响应：

```json
{
  "token": "jwt-token",
  "user": {
    "id": "admin-001",
    "account": "admin001",
    "nickname": "审核管理员",
    "role": "admin"
  }
}
```

后端需要校验账号是否具备管理员角色，避免普通用户通过前端参数伪造进入管理端。

## 5. 菜品投稿审核接口

### GET /api/admin/dish-submissions

查询菜品投稿审核列表。

查询参数：

```text
status=pending|approved|rejected|all
page=1
pageSize=20
keyword=菜品名/食堂/档口/提交人
```

响应：

```json
{
  "list": [
    {
      "id": "submission-001",
      "dishName": "藤椒鸡丝拌面",
      "canteenId": "dongqu",
      "canteenName": "东区餐厅",
      "stallName": "清爽麻香",
      "price": 15,
      "submitter": "2024211001",
      "submittedAt": "2026-05-09 09:20",
      "status": "pending",
      "imageUrl": "https://example.com/uploads/dish.jpg",
      "description": "菜品说明",
      "tags": ["清爽", "拌面"],
      "reason": ""
    }
  ],
  "total": 1
}
```

### POST /api/admin/dish-submissions/{id}/approve

通过菜品投稿。

请求：

```json
{
  "reason": "图片清晰，描述完整，已进入菜品展示候选池。"
}
```

响应：

```json
{
  "id": "submission-001",
  "status": "approved",
  "reason": "图片清晰，描述完整，已进入菜品展示候选池。"
}
```

建议后端通过后同步创建或更新菜品记录，并记录审核人、审核时间。

### POST /api/admin/dish-submissions/{id}/reject

驳回菜品投稿。

请求：

```json
{
  "reason": "图片不清晰，请重新上传。"
}
```

响应：

```json
{
  "id": "submission-001",
  "status": "rejected",
  "reason": "图片不清晰，请重新上传。"
}
```

## 7. 用户侧接口联动

用户发布吐槽时，后端应默认写入 `pending` 状态，不应直接公开展示：

```text
POST /api/rants
```

响应建议返回：

```json
{
  "id": "rant-101",
  "status": "pending",
  "message": "已提交审核，通过后会展示在吐槽墙。"
}
```

吐槽墙公开列表只返回 `approved` 内容：

```text
GET /api/rants?status=approved
```

用户菜品投稿同理，默认创建为 `pending`：

```text
POST /api/dish-submissions
GET /api/my/dish-submissions
```

`GET /api/my/dish-submissions` 需要返回当前用户自己的投稿状态和处理意见。

## 9. 接入建议

+ 后端接口统一返回标准错误结构，例如 `{ "code": "FORBIDDEN", "message": "无管理员权限" }`。
+ 审核接口必须校验管理员 token，不只依赖前端隐藏入口。
+ 审核记录建议落库保存 `auditorId`、`auditedAt`、`reason`、`previousStatus`、`nextStatus`。
+ 列表接口建议支持分页，避免管理端一次性拉取过多内容。
+ 图片上传建议返回稳定 `imageUrl`，不要只保存文件名。
+ 吐槽墙公开接口只返回 `approved`，管理端接口才允许查询 `pending` 和 `rejected`。
